Tự động thuyết minh du lịch ẩm thực theo vị trí thực tế, hỗ trợ đa ngôn ngữ và offline-first.

1. Tổng quan dự án

VinhKhanhTourGuide là hệ thống thuyết minh du lịch ẩm thực thông minh hoạt động theo mô hình Fullstack Client–Server. Hệ thống cho phép quản lý tập trung dữ liệu điểm đến, đồng bộ dữ liệu về ứng dụng di động, tự động phát thuyết minh khi khách du lịch đi vào vùng gần quán và ghi nhận dữ liệu nghe để phục vụ phân tích vận hành.

Tư tưởng cốt lõi: quản lý tập trung ở server, trải nghiệm tự động ở mobile, vẫn hoạt động được khi mạng yếu nhờ cơ chế cache SQLite trên thiết bị.

Bài toán

Khách du lịch khó tiếp cận thông tin món ăn và câu chuyện địa phương theo đúng vị trí thực tế.

Giải pháp

Ứng dụng mobile tự phát hiện vị trí, tự động phát nội dung, hỗ trợ dịch và phát âm bằng TTS.

Lợi ích

Tăng trải nghiệm tham quan, giảm phụ thuộc hướng dẫn viên, tạo dữ liệu phân tích hành vi nghe.

2. Mục tiêu sản phẩm

2.1. Mục tiêu nghiệp vụ

  • Số hóa danh sách quán ăn và nội dung thuyết minh của khu ẩm thực Vĩnh Khánh.
  • Tạo một trải nghiệm tham quan tự động, trực quan, dễ dùng trên điện thoại.
  • Cho phép quản trị viên cập nhật tọa độ, nội dung và bán kính nhận diện tập trung trên web.
  • Thu thập dữ liệu nghe để đánh giá độ quan tâm của khách tại từng địa điểm.

2.2. Mục tiêu kỹ thuật

  • Tổ chức hệ thống theo mô hình 3 lớp dễ mở rộng.
  • Đồng bộ dữ liệu giữa SQL Server trung tâm và SQLite local trên thiết bị.
  • Quét GPS định kỳ để phát hiện vào vùng geofence.
  • Hỗ trợ phát âm thanh đa ngôn ngữ theo ngôn ngữ máy.
  • Gửi analytics log lên server để tổng hợp theo thời gian thực gần đúng.

3. Phạm vi và đối tượng sử dụng

3.1. Đối tượng sử dụng

  • Khách du lịch: dùng ứng dụng mobile để xem bản đồ, nghe thuyết minh, mở chỉ đường.
  • Quản trị viên: dùng Web Admin để CRUD POI và xem thống kê analytics.
  • Người vận hành hệ thống: theo dõi API, dữ liệu log và cơ sở dữ liệu trung tâm.

3.2. Trong phạm vi

  • CRUD điểm đến/quán ăn.
  • Đồng bộ dữ liệu qua endpoint /api/pois.
  • Quét geofence mỗi 5 giây trên app mobile.
  • Cache dữ liệu POI và bản dịch trên SQLite.
  • TTS phát nội dung theo ngôn ngữ máy.
  • Gửi log nghe qua endpoint /api/listeninglogs.
  • Hiển thị thống kê và heatmap trên Web Admin.

3.3. Ngoài phạm vi hiện tại

  • Đăng nhập người dùng cuối và phân quyền phức tạp.
  • Thanh toán, đặt bàn, đặt món.
  • Push notification.
  • CMS nội dung nhiều bước duyệt.
  • Cơ chế AI translation đi qua backend an toàn hoàn chỉnh.

4. Kiến trúc hệ thống 3 lớp

Lớp Thành phần Công nghệ Vai trò
Presentation Layer Mobile App + Web Admin .NET MAUI, ASP.NET Core MVC Hiển thị giao diện cho khách du lịch và quản trị viên.
Application Layer Web API ASP.NET Core Web API Cung cấp dữ liệu POI và tiếp nhận log analytics.
Data Layer SQL Server + SQLite SQL Server, SQLite Lưu trữ tập trung và cache offline trên thiết bị.
Thông số triển khai hiện tại:
- API nội bộ chạy qua cổng 5099
- Android Emulator truy cập host qua IP 10.0.2.2
- Mobile dùng SQLite local để lưu dữ liệu quán và cache bản dịch
- Web Admin ép culture en-US để nhập tọa độ GPS chính xác

Luồng dữ liệu chính

  1. Admin cập nhật POI vào SQL Server.
  2. Mobile gọi API để lấy danh sách POI.
  3. Mobile lưu POI vào SQLite local.
  4. Mobile quét GPS và phát thuyết minh khi vào vùng.
  5. App gửi log nghe về server để analytics tổng hợp.

Nguyên tắc thiết kế

  • Offline-first cho trải nghiệm ngoài hiện trường.
  • Ưu tiên đơn giản, dễ demo và dễ bảo trì đồ án.
  • Luồng dữ liệu rõ ràng giữa Admin → API → Mobile.
  • Tối ưu cho kịch bản một khu ẩm thực với số lượng POI vừa phải.

5. Phân hệ Web Admin (MVC)

Web Admin là giao diện vận hành của hệ thống, cho phép quản trị viên quản lý nội dung điểm đến và theo dõi dữ liệu thống kê người dùng.

5.1. Chức năng chính

  • Quản lý danh sách POI theo mô hình CRUD: Thêm, Xem, Sửa, Xóa.
  • Nhập tọa độ GPS chuẩn dấu chấm thập phân nhờ culture en-US.
  • Quản lý các trường: mã quán, tên quán, ảnh, vĩ độ, kinh độ, mô tả, geofence radius, priority.
  • Xem trang analytics tổng hợp số lượt nghe, thời lượng trung bình và dữ liệu heatmap.

5.2. Vai trò nghiệp vụ

  • Là nơi duy nhất chỉnh sửa dữ liệu POI trung tâm.
  • Đảm bảo dữ liệu đầu vào chính xác để mobile app đồng bộ xuống.
  • Hỗ trợ người vận hành đánh giá quán nào được nghe nhiều nhất.

5.3. Kết quả đầu ra

  • Bảng Poi trên SQL Server luôn phản ánh dữ liệu chính thức của hệ thống.
  • Analytics hiển thị các chỉ số tổng hợp thân thiện với người quản trị.

6. Phân hệ Web API

Web API đóng vai trò cầu nối giữa cơ sở dữ liệu trung tâm và ứng dụng mobile. API hiện tại được thiết kế đơn giản để phục vụ đúng nhu cầu cốt lõi của đồ án.

6.1. Endpoint hiện có

Method Endpoint Mục đích Đầu ra
GET /api/pois Lấy toàn bộ danh sách quán/điểm đến JSON danh sách POI
POST /api/listeninglogs Nhận log nghe từ mobile app Trạng thái lưu thành công/thất bại

6.2. Trách nhiệm của API

  • Truy vấn SQL Server để trả danh sách POI cho mobile.
  • Ghi nhận log nghe, tự gán thời điểm ListenAt từ server.
  • Trả lỗi rõ ràng để phía mobile in log debug khi demo.
  • Tạo một lớp trung gian tách biệt giữa client và database.
Điểm tốt của thiết kế hiện tại: API đã tách được luồng lấy dữ liệu và luồng ghi analytics, giúp mobile không thao tác trực tiếp với SQL Server.

7. Phân hệ Mobile App (MAUI)

Mobile App là phần tương tác trực tiếp với khách du lịch. Đây là trái tim trải nghiệm của hệ thống, chịu trách nhiệm tải dữ liệu, hiển thị bản đồ, quét GPS, phát thuyết minh, cache nội dung và gửi analytics.

7.1. Chức năng chính

  • Xin quyền truy cập vị trí của thiết bị.
  • Tải dữ liệu quán ăn từ SQLite local; trước đó cố gắng đồng bộ từ API.
  • Hiển thị pin các địa điểm trên bản đồ.
  • Hiển thị danh sách quán ăn trong CollectionView.
  • Quét GPS mỗi 5 giây bằng timer.
  • Nếu vào vùng geofence, chọn POI theo Priority rồi đến khoảng cách.
  • Highlight vòng tròn đỏ trên bản đồ cho POI đang phát.
  • Lấy bản dịch từ cache hoặc gọi AI để dịch khi chưa có.
  • Phát âm bằng Text-to-Speech và cho phép dừng phát.
  • Gửi log analytics sau khi nghe xong hoặc bấm dừng.

7.2. Offline-first logic

  1. Khi khởi tạo app, tạo bảng Poi và TranslationCache trong SQLite.
  2. Gọi API để lấy dữ liệu POI mới nhất nếu có mạng.
  3. Nếu gọi thành công, xóa dữ liệu POI cũ và ghi đè dữ liệu mới.
  4. Nếu không có dữ liệu và cũng không gọi được API, nạp bộ dữ liệu backup cứng để tránh app trống.

7.3. Radar và Geofencing

Luật phát hiện POI mục tiêu:
1. App lấy vị trí hiện tại của người dùng.
2. Tính khoảng cách tới từng POI trong danh sách.
3. Nếu khoảng cách ≤ GeofenceRadius, POI được coi là hợp lệ.
4. Nếu có nhiều POI cùng hợp lệ, ưu tiên POI có Priority nhỏ hơn.
5. Nếu trùng priority, chọn POI gần nhất.
6. Áp dụng cooldown để tránh đọc lặp lại quá nhanh.

7.4. Phát thuyết minh

  • Nếu ngôn ngữ thiết bị là tiếng Việt, dùng trực tiếp Description_VN.
  • Nếu khác tiếng Việt, app tìm trong bảng TranslationCache.
  • Nếu cache trống, app gọi AI để dịch rồi lưu lại.
  • TTS của hệ điều hành đọc nội dung và có thể dừng bằng nút Stop.

7.5. Màn hình chính

  • MapPage: hiển thị bản đồ, pins, radar geofence, list quán, nút dừng audio.
  • EateryDetailPage: hiển thị chi tiết quán, phát audio thủ công, mở chỉ đường đi bộ.
  • SplashPage: trang mở đầu đơn giản của ứng dụng.

9. Quy trình xử lý nghiệp vụ

9.1. Quy trình đồng bộ dữ liệu

  1. Mobile app khởi tạo SQLite local.
  2. App gọi API GET /api/pois.
  3. Nếu lấy dữ liệu thành công, app ghi đè bảng Poi local.
  4. Nếu thất bại, app dùng dữ liệu local hoặc bộ backup cứng.

9.2. Quy trình thuyết minh tự động

1. Quét GPS mỗi 5 giây.
2. So sánh khoảng cách với GeofenceRadius của từng POI.
3. Tạo danh sách POI hợp lệ trong vùng.
4. Chọn POI có Priority nhỏ nhất, nếu bằng nhau thì chọn POI gần nhất.
5. Kiểm tra cooldown để tránh phát lặp.
6. Kiểm tra cache bản dịch theo PoiId + LanguageCode.
7. Nếu chưa có, gọi AI dịch từ Description_VN và lưu cache.
8. Phát TTS và vẽ vòng tròn highlight đỏ trên bản đồ.
9. Sau khi kết thúc hoặc dừng giữa chừng, gửi analytics log lên server.

9.3. Quy trình analytics

  1. Mobile sinh hoặc đọc AnonymousSessionId từ Preferences.
  2. App đo thời lượng nghe thực tế bằng stopwatch.
  3. Khi kết thúc, app gọi POST /api/listeninglogs.
  4. Server gán ListenAt và lưu vào SQL Server.
  5. Web Admin gom nhóm theo PoiId để tính tổng lượt nghe và thời lượng trung bình.

11.3. Tiêu chí nghiệm thu

  • Admin tạo một POI mới và POI đó xuất hiện trên mobile sau lần đồng bộ kế tiếp.
  • Mobile hiển thị ít nhất toàn bộ các POI hiện có trong ảnh snapshot dữ liệu.
  • Khi thiết bị đi vào phạm vi 30m của POI, app tự động phát thuyết minh.
  • Nếu có bản dịch cache, app sử dụng lại mà không gọi AI lần nữa.
  • Khi bấm Stop, TTS dừng ngay và log vẫn được gửi về server.
  • Analytics trên Web Admin hiển thị đúng tổng lượt nghe và thời lượng trung bình theo POI.
  • Hệ thống vẫn hiển thị được dữ liệu local ngay cả khi API không truy cập được.

12. Rủi ro, hạn chế và hướng phát triển

12.1. Rủi ro và hạn chế hiện tại

Hạn chế kỹ thuật cần ghi nhận trong đồ án: hiện tại logic dịch AI đang được gọi trực tiếp từ mobile app, chưa đi qua backend, nên chưa phải mô hình tối ưu nhất về bảo mật khóa API và kiểm soát chi phí.
  • Khóa AI đang nằm phía client nên có rủi ro bảo mật.
  • Logic geofence còn nằm nhiều trong UI thay vì tách hẳn thành service độc lập.
  • Analytics mới ở mức cơ bản, chưa có dashboard nhiều chiều.
  • Chưa có xác thực quản trị viên nâng cao.
  • Dữ liệu backup cứng trong app chỉ phù hợp giai đoạn demo hoặc fallback.

12.2. Hướng phát triển tiếp theo

  • Đưa translation AI về backend để bảo vệ API key.
  • Tách geofence engine thành service riêng để dễ test.
  • Bổ sung dashboard thời gian thực, biểu đồ theo ngày và theo ngôn ngữ.
  • Bổ sung đăng nhập và phân quyền cho quản trị viên.
  • Cho phép quản lý nhiều khu du lịch thay vì chỉ khu Vĩnh Khánh.
  • Tối ưu chiến lược cache với composite index hoặc unique theo PoiId + LanguageCode.
